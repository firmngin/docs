Firmware management in firmngin tracks what version is available, which devices should receive it, and how rollout progress is observed.

## Firmware concepts

| Concept | Meaning |
| --- | --- |
| Artifact | The compiled firmware file or binary produced by a build pipeline. |
| Version | Human-readable release identifier such as `1.4.2`. |
| Policy | Rules that decide which devices are eligible for a rollout. |
| Batch | A controlled group of target devices in a rollout. |
| Device result | Per-device install state, logs, and errors. |

## Artifact versus policy

Keep firmware artifacts separate from rollout policies.

- Artifact upload answers: "What binary can be installed?"
- Rollout policy answers: "Which devices should receive it, when, and under what constraints?"

This separation matters because manual upload, CI/CD upload, and fleet rollout governance do not have the same permissions or review process.

## Release workflow

1. Build firmware from a trusted source.
2. Upload or register the artifact.
3. Create a release version.
4. Define targeting policy.
5. Start with a small batch.
6. Monitor install progress and logs.
7. Expand rollout only after the previous batch is healthy.

## CI/CD path

For automated releases, use the dedicated CI/CD upload path configured for firmware artifacts. Do not widen manual upload behavior to simulate CI/CD permissions.
